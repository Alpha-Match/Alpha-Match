# AI 협업을 위한 효율적인 문서 관리 가이드

이 문서는 AI 에이전트(Gemini)와의 협업 효율을 극대화하기 위한 `GEMINI.md` 최적화 및 프로젝트 문서 관리 전략을 다룹니다.

---

## 1. `GEMINI.md` 최적화를 통한 메모리 효율화

`GEMINI.md` 파일은 AI 에이전트가 대화의 매 턴마다 참고하는 **핵심 컨텍스트**입니다. 이 파일의 크기와 내용은 에이전트의 응답 속도, 정확성, 그리고 비용에 직접적인 영향을 미칩니다. 따라서 이 파일을 효율적으로 관리하는 것은 매우 중요합니다.

### 🧠 핵심 원칙: "AI의 두뇌에 무엇을 남길 것인가?"

- **Global & Stable**: 프로젝트 전반에 걸쳐 적용되며, 자주 바뀌지 않는 최상위 원칙과 구조를 명시합니다.
- **Concise & Essential**: 세부 구현보다는 핵심적인 설계 사상, 기술 스택, 반드시 지켜야 할 규칙(Do/Don't) 위주로 요약합니다.
- **Discoverable vs. Given**: 파일 내용, 특정 변수명 등 AI가 `ls`, `grep`, `read_file` 같은 도구로 **찾을 수 있는 정보**는 `GEMINI.md`에 넣지 않습니다. 대신, AI가 스스로 **발견할 수 없는** 추상적인 아키텍처, 결정 배경, 주요 제약사항을 알려주는 데 집중합니다.

### ✅ 무엇을 포함해야 하는가? (What to Include)

- **Tech Stack**: 핵심 프레임워크, 언어, 주요 라이브러리 목록 (예: Next.js, Spring WebFlux, pgvector)
- **High-Level Architecture**: 전체 시스템의 주요 구성 요소와 그들 간의 데이터 흐름 (예: Frontend -> BFF -> API Server -> DB)
- **Project Structure Convention**: 모듈, 디렉터리 구조에 대한 최상위 규칙 (예: `Backend/Api-Server`, `Frontend/Front-Server`)
- **Critical 'Do Not' Rules**: 절대 수행해서는 안 되는 작업 (예: 특정 디렉토리 접근 금지, 특정 라이브러리 사용 금지)
- **User Preferences**: 사용자가 명시적으로 요청한 작업 스타일, 커밋 메시지 형식 등 (이 내용은 `save_memory` 도구를 통해 장기기억으로 관리하는 것이 더 효율적일 수 있습니다.)

### ❌ 무엇을 제외해야 하는가? (What to Exclude)

- **File Contents**: `read_file`로 읽을 수 있는 모든 소스 코드, 설정 파일 내용
- **Detailed Logic**: 특정 함수의 상세한 구현 로직, 알고리즘
- **Volatile Information**: 자주 변경되는 기획 내용, 임시 작업 계획, 버그 티켓 목록
- **Directory Listings**: `ls` 명령어로 확인할 수 있는 파일 및 디렉토리 목록
- **Obvious Information**: 일반적인 프로그래밍 상식이나 프레임워크의 표준 사용법

### 💡 모듈별 컨텍스트 분리

프로젝트가 커지면 루트 `GEMINI.md`는 최상위 컨텍스트만 유지하고, 각 하위 프로젝트(e.g., `Backend/Batch-Server`)에 별도의 `GEMINI.md` 또는 `CLAUDE.md`를 두어 해당 모듈에만 국한된 컨텍스트를 제공하는 것이 효율적입니다.

## 2. 체계적인 문서 관리 전략

효율적인 문서 관리는 AI와의 협업뿐만 아니라, 사람 개발자 간의 협업과 프로젝트 유지보수성을 높이는 핵심 요소입니다.

### 🗺️ 문서의 종류와 역할

1.  **`GEMINI.md` / `CLAUDE.md` (for AI)**
    - **목적**: AI 에이전트의 컨텍스트 제공
    - **위치**: 프로젝트 루트, 주요 하위 모듈 루트
    - **특징**: 간결성, 핵심 원칙 위주, AI가 발견할 수 없는 추상적 정보 제공

2.  **`README.md` (for Human)**
    - **목적**: 개발자를 위한 프로젝트 시작 가이드
    - **위치**: 프로젝트 루트, 주요 하위 모듈 루트
    - **내용**: 프로젝트 개요, 기술 스택, 설치 및 실행 방법, 테스트 방법 등

3.  **`/docs` 디렉토리 (Detailed Documentation)**
    - **목적**: 상세 설계, 의사결정 기록, 아키텍처 다이어그램 등 심층적인 정보 아카이빙
    - **위치**: 프로젝트 루트, 필요시 하위 모듈 내부
    - **내용**:
        - API 명세서
        - DB 스키마 다이어그램 (ERD)
        - Mermaid를 활용한 시스템 흐름도, 시퀀스 다이어그램
        - 아키텍처 결정 기록 (ADR)
        - 트러블슈팅 가이드

### 🎨 시각화와 구조화

- **Mermaid 활용**: 복잡한 데이터 흐름, 아키텍처, 요청/응답 시퀀스는 Mermaid 다이어그램으로 시각화하여 AI와 개발자 모두의 이해를 돕습니다. (사용자 선호사항)
- **구조화된 형식**: Notion Markdown 형식에 맞춰 헤딩(`##`, `###`), 리스트, 테이블을 적극적으로 사용하여 가독성을 높입니다.

### 🔄 문서의 최신성 유지

- 코드가 변경될 때 관련 문서를 함께 수정하는 것을 원칙으로 합니다.
- 특히 API 변경, 아키텍처 수정 등 중요한 변경사항은 `/docs` 와 `GEMINI.md`에 반드시 반영하여 AI와 동료 개발자에게 최신 정보를 제공해야 합니다.
